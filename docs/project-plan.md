# TurboInfer 项目计划

## 目标

构建一个迷你 LLM 推理服务引擎，展示现代推理基础设施使用的核心性能优化思路。项目规模要小到可以完成，但深度要足以在 AI Infra 面试中讨论。

## 12 周路线图

### 第 1-2 周：基线推理

**范围**：

- 加载 tokenizer 和因果语言模型
- 实现朴素的 Prefill + Decode 流程
- 支持简单的单请求生成
- 记录基线延迟和内存使用

**预期输出**：

- TTFT、TPOT、tokens/s、内存的基线数据
- 清晰解释朴素解码为何低效

### 第 3-4 周：KV Cache

**范围**：

- 分离 Prefill 和 Decode 路径
- 在自回归生成过程中启用 KV Cache
- 对比有缓存和无缓存的生成效果

**预期输出**：

- 前后延迟对比表
- 计算节省和内存开销的解释
- Prefill vs Decode 的性能分析笔记

### 第 5-6 周：连续批处理

**范围**：

- 添加请求队列
- 将多个解码请求批处理在一起
- 支持生成过程中的新请求到达
- 定义调度器行为和公平性策略

**预期输出**：

- 并发请求下的吞吐和延迟对比
- P50/P95/P99 延迟指标
- 吞吐-延迟权衡的笔记

当前第一步实现见 [optimizations/continuous-batching.md](optimizations/continuous-batching.md)。该版本先验证固定 batch decode，后续再加入动态请求到达和尾延迟统计。

### 第 7-8 周：分页 KV Cache

**范围**：

- 引入基于块的 KV Cache 分配
- 跟踪请求到块的映射
- 减少内存碎片
- 文档说明与简单连续 KV Cache 的区别

**预期输出**：

- 内存使用对比
- 最大并发请求数对比
- 解释分页 KV Cache 对服务的意义

### 第 9-10 周：Triton 内核

**范围**：

- 用 Triton 内核替换选定的 PyTorch 操作
- 从小型独立的内核开始，如 RMSNorm 或 RoPE
- 分析内核级影响

**预期输出**：

- 与 PyTorch 的正确性验证
- 每个内核的延迟对比
- 解释自定义内核何时有效、何时无效

### 第 11 周：量化

**范围**：

- 评估 W8 和 W4 仅权重量化
- 对比内存使用、吞吐和质量影响
- 记录 T4 与更新 GPU 的硬件限制差异

**预期输出**：

- 内存和速度对比
- 量化权衡的笔记
- 模型兼容性说明

### 第 12 周：服务与基准测试报告

**范围**：

- 暴露 OpenAI 兼容的生成端点
- 添加基准测试脚本和标准配置
- 在相同模型、提示词长度、输出长度、并发下与 vLLM 对比
- 撰写最终项目报告

**预期输出**：

- 可复现的基准测试表
- 最终架构图
- 可用于面试的项目总结

## 非目标

- 完全重写 vLLM
- 支持所有模型架构
- 第一版支持多节点分布式推理
- 初始阶段达到生产级可靠性

## 最终叙事

项目应呈现为从朴素推理到小型服务引擎的量化之旅。每个优化都应有原因、实现说明和基准测试结果。
